ANTG001 请简述消息中间件(message broker)的工作原理？它大概有哪些参与者，每个参与者做了什么事情?

消息中间件是一种分布式系统组件，用于在不同应用程序或服务之间传递消息，实现异步通信和解耦
```
Producer → Message Broker → Consumer
    ↓           ↓              ↓
  发送消息    存储/路由消息    接收消息

┌─────────────┐    1.发送消息     ┌─────────────────┐    3.投递消息    ┌─────────────┐
│  Producer   │ ──────────────→  │ Message Broker  │ ──────────────→ │  Consumer   │
│             │                  │                 │                 │             │
│ - 业务逻辑   │    2.存储确认     │ - 消息存储       │    4.处理确认    │ - 业务处理   │
│ - 消息创建   │ ←──────────────  │ - 路由分发       │ ←────────────── │ - ACK确认   │
└─────────────┘                  │ - 持久化         │                 └─────────────┘
                                 └─────────────────┘
```

生产者 (Producer/Publisher)
- 创建和发送消息到消息中间件
- 指定消息的目标队列或主题
- 处理发送确认和错误

消息代理 (Message Broker)
- 接收来自生产者的消息
- 存储消息（持久化）
- 路由消息到正确的队列/主题
- 管理消息的生命周期
- 提供消息传递保证（至少一次、恰好一次等）

消费者 (Consumer/Subscriber)
- 连接到消息中间件
- 订阅特定队列或主题
- 接收和处理消息
- 发送消息确认（ACK）

消息传递模式
点对点模式 (Point-to-Point)
发布订阅模式 (Publish-Subscribe) topic 

业务价值：
- 解耦 : 系统间不直接依赖
- 异步 : 提升响应速度和吞吐量
- 可靠 : 消息不丢失，支持重试

场景：
- 订单系统 : 下单后异步发送短信、邮件（发通知、事件驱动）
- 秒杀场景 : 削峰，避免数据库压垮 （削峰）
- 数据同步 : 用户信息变更同步到各个系统 （最终一致性）

使用MQ的场景 :

- ✅ 不需要立即响应
- ✅ 允许最终一致性
- ✅ 一对多通知
- ✅ 需要削峰填谷

- ✅ 需要立即响应
- ✅ 强一致性要求
- ✅ 事务性操作
- ✅ 查询类操作

实现选型：
- Apache Kafka : 高吞吐量，适合大数据场景
- RabbitMQ : 功能丰富，支持多种协议
- Apache RocketMQ : 阿里开源，金融级可靠性
- Apache Pulsar : 云原生，存储计算分离



JITU002 Mq是一种异步并发的理念，如果下游积压了消息，怎么去处理才能保证时效性？生产数据变不了或者不可控，从消费者方面去考虑

当下游积压消息时，需要从 消费能力提升 、 处理策略优化 、 架构设计改进 三个维度来保证时效性。

消费能力提升策略
- 水平扩容 - 增加消费者实例
- 垂直优化 - 提升单实例处理能力 （性能优化：批量消费、异步处理、并行处理）

处理策略优化
- 消息优先级处理(分优先级)  高优先级队列 紧急tag VIP用户（按业务分类）
- 分层处理策略 （分层次）  分类：实时、准实时、延时跑批（按架构分类）
- 智能降级机制 （正常逻辑 →降级逻辑）  思考：怎么实现？

架构设计优化
- 多级队列： 实时队列、普通队列、延迟队列
- 背压（back pressure）控制机制 （思考）
- 消息路由优化 （思考）

监控与自适应调优
- 关键指标监控：queueDepth, cusumeRate, produceRate
- 自适应调优 (思考)


> 你说的【处理策略优化 ：分层处理策略】和【 架构设计优化 多级队列： 实时队列、普通队列、延迟队列】有什么区别？
分层处理策略 是 逻辑层面(这是在同一个消费者内部的分层处理) 的处理方式，
 多级队列架构 是 物理层面(消息在生产时就被分发到不同的物理队列) 的基础设施。


```
分层处理策略适用场景适用于：
✅ 消息类型判断复杂
✅ 处理逻辑经常变化
✅ 队列数量受限
✅ 需要动态调整策略

示例：
订单消息 → 单一队列 → 消费者内部判断
                    ├── VIP订单：立即处理
                    ├── 普通订单：正常处理
                    └── 预订单：延迟处理

多级队列架构适用场景
适用于：
✅ 消息类型明确
✅ 需要资源隔离
✅ 不同类型消息量差异大
✅ 需要独立扩缩容

示例：
订单系统 → 路由器 → ├── 支付队列 → 金融级消费者
                  ├── 通知队列 → 通知消费者
                  └── 日志队列 → 批量消费者
 ```

演进：
 阶段1：单队列 + 分层处理
       ↓ (业务复杂度增加)
阶段2：多级队列 + 简单处理
       ↓ (精细化需求)
阶段3：多级队列 + 分层处理
