观察者模式

观察者模式（Observer Pattern）是一种行为型设计模式，用于定义对象间的一对多依赖关系，当一个对象（主题/Subject）的状态发生变化时，所有依赖它的观察者（Observers）都会自动收到通知并更新。


“观察者模式是一种行为型设计模式。它定义了对象间的一对多依赖：有一个主题对象（Subject），它维护一个观察者列表。当主题的状态改变时，会通知所有观察者自动更新。

核心组件包括：

- Subject：维护观察者列表，提供添加/移除观察者和通知方法。
- Observer：定义更新接口，当收到通知时执行更新。
优点：松耦合，支持广播通信。
缺点：可能导致意外更新或性能问题如果观察者太多。

例如，天气站（Subject）更新数据时，通知多个显示屏（Observers）刷新。”

懒汉、饿汉、双检锁

单例模式的线程安全问题主要出现在懒汉式实现中，多线程可能同时创建实例，导致不止一个对象。为了解决，我推荐几种方式，但既然提到双检锁（Double-Checked Locking, DCL），我先解释为什么用它。

简单来说，双检锁是为了优化性能。传统的同步方法（如在getInstance上加synchronized）虽然安全，但每次调用都加锁，即使实例已存在，这在高并发下效率低下。

双检锁的思路是：先无锁检查实例是否为空（第一检），如果不为空直接返回；如果为空，才加锁，然后在锁内再检查一次（第二检），确认后创建。这样，大部分时间避免了锁开销，只在首次创建时同步。

代码上，需要用volatile修饰实例变量，防止Java内存模型的指令重排序，确保实例完全初始化。举例：

private static volatile Singleton instance;

public static Singleton getInstance() {
if (instance == null) {
synchronized (Singleton.class) {
if (instance == null) {
instance = new Singleton();
}
}
}
return instance;
}

不过，双检锁有点复杂，JDK早期有bug。现在的最佳实践是静态内部类或枚举，更简单安全。比如枚举：public enum Singleton { INSTANCE; }，JVM保证线程安全。

策略模式，哪几部分组成

策略模式（Strategy Pattern）是一种行为型设计模式，用于定义一系列算法，将每个算法封装起来，使它们可互换。核心目的是将算法的使用与算法的实现分离，符合开闭原则（对扩展开放，对修改关闭）。

策略模式由三大部分组成：

1. Strategy（策略接口） ：定义所有算法的公共接口。
2. ConcreteStrategy（具体策略类） ：实现Strategy接口的具体算法变体。
3. Context（上下文类） ：持有Strategy引用，根据需要切换并执行具体策略。

策略模式和模板模式有什么区别

策略模式（Strategy Pattern）和模板模式（Template Method Pattern）都是行为型设计模式，用于处理算法变体，但核心区别在于实现方式和灵活性：

策略模式通过**组合（composition） 动态替换整个算法，强调运行时可互换；

模板模式通过 继承（inheritance）**定义算法骨架，子类只自定义特定步骤，强调固定流程。

- 策略模式 ：适合算法完全替换的场景，如支付方式切换。
- 模板模式 ：适合算法框架固定、仅变异步骤的场景，如数据处理流程。

你写了你熟悉设计模式，能大概讲一下适配器模式吗？


"适配器模式是一种结构型设计模式，用于解决接口不兼容问题。它的核心作用是让原本不兼容的两个接口能够协同工作，就像电源适配器一样。

适配器模式有三个核心组件：

1. **Target（目标接口）**：客户端期望的接口
2. **Adaptee（被适配者）**：现有的、需要被适配的类
3. **Adapter（适配器）**：实现Target接口，内部持有Adaptee实例，负责转换调用

实现方式有两种：
- **对象适配器**：通过组合，适配器持有被适配者实例
- **类适配器**：通过继承，适配器继承被适配者（Java中较少用，因为单继承限制）

典型应用场景：
- 第三方库集成：比如集成不同的支付SDK
- 数据格式转换：XML转JSON
- 遗留系统改造：新系统调用老接口

优点是解耦、复用现有代码；缺点是增加系统复杂度。

举个简单例子：我们有个MediaPlayer只能播放MP3，但现在要播放MP4。我们创建一个MediaAdapter，内部调用AdvancedMediaPlayer的playMp4方法，对外提供统一的play接口。"

**记忆要点：**
- 核心：接口转换，让不兼容变兼容
- 结构：Target + Adaptee + Adapter
- 场景：第三方集成、格式转换、遗留系统
- 类比：电源适配器
