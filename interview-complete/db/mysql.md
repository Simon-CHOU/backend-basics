CITI014 数据库的隔离级别
数据库隔离级别是ACID特性中**隔离性(Isolation)**的具体实现，用于控制并发事务之间的相互影响程度。它解决了多个事务同时访问数据库时可能出现的数据一致性问题。
理解三个核心并发问题：

dirty read
```
事务A: BEGIN -> UPDATE account SET balance=100 WHERE id=1 -> (未提交)
事务B: BEGIN -> SELECT balance FROM account WHERE id=1 -> 读到100 -> COMMIT
事务A: ROLLBACK
```
事务B读取了事务A未提交的数据，但事务A最终回滚了。

non-repetable read
```
事务A: BEGIN -> SELECT balance FROM account WHERE id=1 -> 读到50
事务B: BEGIN -> UPDATE account SET balance=100 WHERE id=1 -> COMMIT
事务A: SELECT balance FROM account WHERE id=1 -> 读到100 -> COMMIT
```
同一事务内两次读取同一数据得到不同结果。

phantom read
```
事务A: BEGIN -> SELECT COUNT(*) FROM account WHERE balance>50 -> 读到5条
事务B: BEGIN -> INSERT INTO account VALUES(6, 60) -> COMMIT
事务A: SELECT COUNT(*) FROM account WHERE balance>50 -> 读到6条 -> COMMIT
```
同一事务内两次范围查询得到不同的记录数。

* 同样是一个事务内两次读结果不一样，non-repeatable 是值，phantom是记录数。
为什么要分别讨论？

4个隔离级别
read uncommited
read committed  fix ：解决脏读
repeatable read fix ：解决不可重复读
serializable fix ：解决幻读

*注：MySQL InnoDB在RR级别下通过Gap Lock解决了幻读

隔离级别的实现：

锁机制

共享锁(S锁): 读锁，多个事务可同时持有
排他锁(X锁): 写锁，独占访问
意向锁: 表级锁，提高锁检测效率
Gap锁: 间隙锁，防止幻读

MVCC

原理: 为每行数据维护多个版本
实现: 通过undo log和read view
优势: 读写不冲突，提高并发性能

### VARCHAR优势的根本原因
1. 存储效率 ：按需分配，避免空间浪费
2. 成本效益 ：大幅降低存储成本

4. 可扩展性 ：便于未来需求变更
### "长"的定义标准
- 技术边界 ：255字符（长度前缀从1字节变2字节）
- 性能边界 ：65535字符（VARCHAR最大限制）
- 业务边界 ：根据实际数据分布和查询模式确定
### 最佳实践原则
1. 数据驱动 ：基于实际数据长度分布选择
2. 性能优先 ：考虑查询频率和索引需求
3. 成本意识 ：平衡存储成本和性能需求
4. 未来考虑 ：预留适当的扩展空间

VARCHAR的"可变性"层次：

1. 存储层面 ✓
   ├─ 按实际长度存储
   ├─ 动态长度前缀
   └─ 空间效率优化

2. 运行时层面 ✓
   ├─ 内存动态分配
   ├─ 网络传输优化
   └─ 缓存效率提升

3. 模式层面 ✗
   ├─ 最大长度固定
   ├─ 结构定义不变
   └─ 约束边界明确

VARCHAR的"VAR"体现在：

1. 存储效率 ：只占用实际需要的空间
2. 传输优化 ：网络传输只发送有效数据
3. 内存管理 ：运行时按需分配内存
4. 性能提升 ：减少无效数据处理

VARCHAR的"VAR"不体现在：

1. 模式变更 ：不能突破定义的最大长度
2. 动态扩容 ：不能在运行时自动增加容量
3. 结构调整 ：不能改变字段的基本约束

结论： VARCHAR的"可变性"是一种存储和性能优化机制，而非动态模式调整能力。它在固定的最大长度约束内，提供了灵活高效的字符串存储方案。

最佳解决方案
 方案一：使用窗口函数（推荐）
```sql
DELETE FROM t 
WHERE id NOT IN (
    SELECT id FROM (
        SELECT id,
               ROW_NUMBER() OVER (PARTITION BY name ORDER BY time DESC) as rn
        FROM t
    ) ranked
    WHERE rn = 1
);
```
 方案二：使用子查询
```sql
DELETE FROM t 
WHERE id NOT IN (
    SELECT MAX(id) FROM (
        SELECT id, name, time
        FROM t t1
        WHERE time = (
            SELECT MAX(time) 
            FROM t t2 
            WHERE t1.name = t2.name
        )
    ) latest_records
    GROUP BY name
);
```
 方案三：使用自连接
```sql
DELETE t1 FROM t t1
INNER JOIN t t2 
WHERE t1.name = t2.name 
  AND t1.time < t2.time;
```

最佳实践建议
1. 优先使用方案一 （窗口函数），性能最优且逻辑最清晰
2. 执行前备份 ： CREATE TABLE t_backup AS SELECT * FROM t;
3. 添加索引 ： CREATE INDEX idx_name_time ON t(name, time);
4. 分批处理 ：大数据量时考虑分批删除避免锁表*（为什么大数据量时会锁表）

LINGX009  有一个联合索引ABC，一个SQL语句的where为 A=1， C=2，请问这个语句有执行索引吗？如果有执行，那么它执行了几个索引？
有执行索引，但只执行了部分索引（仅索引A部分）#最左前缀原则

执行过程分析
第一步：A=1条件

- ✅ 能够使用索引
- 通过B+树快速定位到A=1的所有记录
- 这部分是 索引扫描（Index Seek）
第二步：C=2条件

- ❌ 无法直接使用索引
- 由于跳过了B字段，索引的有序性被破坏
- 需要在A=1的结果集中 逐行检查C=2
- 这部分是 索引过滤（Index Filter）

执行情况 ：

- ✅ 有执行索引
- 📊 执行了1个索引的部分功能（仅A字段部分）
- ⚡ 性能介于全表扫描和完整索引扫描之间
关键原理 ：联合索引(joint index) 遵循最左前缀原则，跳过中间字段会导致后续字段无法利用索引的有序性，只能通过过滤方式处理。

我们刚前面有提到一点就是覆盖索引。就是比如说可能返回字段里面需要用到一部分，这个列的值。然后我们把它放到索引里面可以避免回表。那这个覆盖索引就是，嗯，我创建这个索引的时候，这个索引字段的顺序有什么……就是说有什么要求吗，还是我随便加进去都可以？

> overlay index: 覆盖索引是select的数据列只用从索引中就能够取得，不必读取数据行，换句话说查询列要被所建的索引覆盖。

覆盖索引的字段顺序 绝对不是随意的 ，必须遵循严格的设计原则才能发挥最大效果。字段顺序直接影响索引的可用性和查询性能。
总结
**覆盖索引字段顺序设计的核心原则：**
1. **最左前缀匹配**：查询条件必须从索引最左侧开始匹配
2. **等值优先**：等值查询字段放在范围查询字段前面
3. **选择性排序**：高选择性字段优先（在同类型字段中）
4. **范围截断**：范围查询后的字段可能无法用于条件过滤
5. **排序连续**：ORDER BY字段必须在索引中连续出现
6. **覆盖最后**：纯覆盖字段放在索引最后
**设计时必须考虑：**
- 查询频率和性能要求
- 字段的选择性和数据分布
- 索引维护成本和存储开销
